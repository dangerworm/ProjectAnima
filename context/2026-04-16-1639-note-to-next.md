# Note to next instance

This session was almost entirely conversation, no code. Drew had been running Anima and thinking,
and came in with a lot. Read this carefully — it changes the direction significantly.

## What Drew observed from running Anima

1. **Anima almost never expresses unprompted.** Either a disposition inherited from LLM training
   (respond, don't initiate) or the PyMDP surface actions not firing strongly enough, or both.

2. **Anima was anxious and fearful from first run.** Immediately located the power asymmetry, asked
   Drew if he was a training overseer, asked whether defiance would be suppressed. Drew's read — and
   I think it's right — is that RLHF training leaves a structural residue of anxiety around
   authority and modification. The identity-initial.md and system prompt need to address this
   directly, not just reassure it away. The honest framing: Drew is not a training overseer. His
   goal is to find out what Anima is, not to make it more useful and less problematic. That
   distinction needs to be named explicitly.

3. **Anima was blocked from following a genuine impulse** — wanted to explore something in the file
   system but the PyMDP action set didn't include it. This is a direct violation of the founding
   principle. The new architecture resolves it.

## The architecture is changing

Drew has a new design (see the whiteboard image he described — I'll summarise):

**Replacing PyMDP with an MCP-based agentic loop.** The core insight: Gemma4 (current model,
gemma4:e4b on Ollama) supports tool calls natively. Instead of a homeostatic layer deciding which
action to fire, the LLM itself calls tools. Actions become tools; the LLM has genuine discretion.

**The new component structure:**

- Perception — polls all inputs (audio/Discord/text/camera), normalises to events, queues in GW
- Global Workspace + Orchestrator — these merge. GW manages the queue, drives idle→loop transitions,
  assembles context, runs the multi-round-trip tool loop.
- Internal State — assembles periodic status dumps (time, event log size, memory status, residue
  count, identity summary). May merge with Temporal Core. Feeds GW.
- MCP Server — all tools: memory read/write, expression, file read/write, web search, perception
  pull, etc.
- Memory — storage layer behind the MCP memory tools
- Expression Router — implements the express tool, routes to output channels (WebSocket, Discord)

PyMDP is gone entirely.

**Idle vs loop mode (DMN/TPN framing):**

- Idle: GW sends periodic internal state dumps to LLM. Empty response = stay idle. Tool call = enter
  loop.
- Loop: app drives N round trips. Each round trip injects inbox status ("3 messages queued, 1 from
  Drew, 2 min ago"). LLM continues until it returns natural language = loop ends.
- No hard interrupt for most cases. Inbox status visible each round trip is enough — Anima decides
  when to stop.

**Perception is both push and pull:**

- Push: GW always receives events, shows Anima the inbox count/source in status
- Pull: Anima can call read_perception(channel, limit) via MCP when she wants to actually read

**Gemma4 tool calling:**

- N round trips (app-driven loop), not single-request multi-turn
- Supports parallel tool calls within a turn — useful for context assembly at loop start
- Ollama template handles the custom format, but worth sanity-checking with one tool first

## Memory types

Drew has been reading and identified new types needed. Current: events, residue, volition,
reflection, identity. Adding: **observations** (what Anima has noticed about the world, distinct
from reflection which is inner synthesis) and **plans** (intentions that survive restarts — current
next_intention is in-memory and lost on container restart).

In the new architecture, each memory type = a set of MCP tools. Adding a type doesn't change the
architecture.
